## 概述

为电竞陪玩平台增加房间语音功能，满足两个核心场景：

- **场景一**：用户下单后，自动创建语音房间，买家和陪玩可以直接进入房间实时交流
- **场景二**：俱乐部内提供语音大厅，成员可进入多频道语音房间，在"陪玩推荐"频道浏览并挑选陪玩

房间语音支持多个频道（如大厅、LOL、王者荣耀、陪玩推荐、闲聊等），用户可在频道间自由切换。

## 技术选型

### 实时语音方案：WebRTC + Pusher 信令

平台已有 Pusher（实时消息推送），可以复用做 WebRTC 信令层，无需引入额外三方服务。语音传输走 P2P 或 SFU 模式。

| 层级 | 选型 | 理由 |
|------|------|------|
| 信令 | **Pusher**（已有） | 复用现有基础设施，负责 SDP/ICE 协商、频道事件广播 |
| 媒体传输 | **WebRTC** | 浏览器原生支持，P2P 直连延迟低 |
| 多路混音 | **SFU 模式升级预留**（Agora / LiveKit） | 初期 3-8 人房间 P2P mesh 可接受；超过 5 人建议引入 SFU |
| 前端 WebRTC | 自封装 `useVoiceRoom` Hook | 管理 mic/camera track、peer connections、静音/闭麦状态 |
| 音频处理 | AudioContext API | 音量检测（说话状态指示）、音频可视化 |

### 推荐初期方案：P2P Mesh（轻量，零成本）

- 2-8 人房间：P2P 直连，WebRTC mesh
- 信令通过 Pusher channel 传递，无需新服务器
- 后续可平滑升级到 LiveKit 或 Agora 的 SFU 模式

### 关键技术栈

```
WebRTC (getUserMedia + RTCPeerConnection + RTCDataChannel)
  ↓ 信令
Pusher (presence channel + private channel)
  ↓ 状态管理
Zustand (voiceRoom store)
  ↓ 前端框架
Next.js 14 + React 18 + Tailwind CSS
```

## 数据模型设计

### Prisma Schema 新增 Model

```prisma
// 语音房间
model VoiceRoom {
  id          String   @id @default(cuid())
  name        String                           // 房间名称
  type        String   @default("order")       // order | club
  orderId     String?                          // 关联订单（下单后自动创建）
  clubId      String?                          // 关联俱乐部
  status      String   @default("active")      // active | closed
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  channels    VoiceChannel[]
  members     VoiceRoomMember[]

  @@index([orderId])
  @@index([clubId])
  @@index([status])
}

// 语音频道（一个房间多个子频道）
model VoiceChannel {
  id          String   @id @default(cuid())
  roomId      String
  name        String                            // 频道名称，如"大厅"、"LOL陪玩"、"王者荣耀"
  sortOrder   Int      @default(0)             // 排序
  maxUsers    Int      @default(10)            // 频道人数上限
  createdAt   DateTime @default(now())

  room         VoiceRoom         @relation(fields: [roomId], references: [id], onDelete: Cascade)
  participants VoiceParticipant[]

  @@index([roomId])
}

// 房间成员
model VoiceRoomMember {
  id          String   @id @default(cuid())
  roomId      String
  userId      String
  role        String   @default("listener")     // host | speaker | listener
  isMuted     Boolean  @default(false)          // 麦克风静音
  isDeafened  Boolean  @default(false)          // 扬声器静音
  joinedAt    DateTime @default(now())
  leftAt      DateTime?

  @@unique([roomId, userId])
  @@index([userId])
}

// 频道参与者（当前在哪个频道）
model VoiceParticipant {
  id          String   @id @default(cuid())
  channelId   String
  userId      String
  roomId      String
  joinedAt    DateTime @default(now())

  channel     VoiceChannel @relation(fields: [channelId], references: [id], onDelete: Cascade)

  @@unique([channelId, userId])
  @@index([userId])
  @@index([roomId])
}
```

## API 路由设计

### 房间管理

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/voice-rooms` | `GET` | 获取活跃房间列表（支持 clubId/orderId 筛选） |
| `/api/voice-rooms` | `POST` | 创建房间（body: `{ type, orderId?, clubId?, name }`） |
| `/api/voice-rooms/[id]` | `GET` | 房间详情（频道列表 + 在线成员 + 各频道人数） |
| `/api/voice-rooms/[id]` | `DELETE` | 关闭房间（房主/管理员） |

### 成员管理

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/voice-rooms/[id]/join` | `POST` | 加入房间 |
| `/api/voice-rooms/[id]/leave` | `POST` | 离开房间 |
| `/api/voice-rooms/[id]/members` | `GET` | 房间在线成员列表 |
| `/api/voice-rooms/[id]/mute` | `POST` | 更新成员静音状态 |

### 频道管理

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/voice-rooms/[id]/channels` | `GET` | 获取房间下所有频道及人数 |
| `/api/voice-rooms/[id]/channels` | `POST` | 创建频道（房主/管理员） |
| `/api/voice-rooms/[id]/channels/[channelId]` | `DELETE` | 删除频道 |
| `/api/voice-channels/[channelId]/join` | `POST` | 加入/切换到指定频道 |
| `/api/voice-channels/[channelId]/leave` | `POST` | 离开频道 |

### WebRTC 信令

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/voice-signaling/token` | `GET` | 生成 Pusher presence channel 认证 token |
| `/api/voice-signaling/sdp` | `POST` | 转发 SDP Offer/Answer（Pusher 服务端 relay） |
| `/api/voice-signaling/ice` | `POST` | 转发 ICE Candidate |

## Pusher 事件设计

复用现有 Pusher 实例，Channel 命名空间新增 `voice-` 前缀避免与聊天冲突。

| Channel | Event | Payload | 说明 |
|---------|-------|---------|------|
| `presence-voice-room-{roomId}` | `member-added` | `{ userId, name, avatar, channelId }` | 成员加入房间 |
| `presence-voice-room-{roomId}` | `member-removed` | `{ userId }` | 成员离开房间 |
| `presence-voice-room-{roomId}` | `client-switch-channel` | `{ userId, fromChannelId, toChannelId }` | 用户切换频道 |
| `presence-voice-room-{roomId}` | `client-mute-changed` | `{ userId, isMuted, isDeafened }` | 静音状态变化 |
| `private-voice-signal-{userId}` | `rtc-offer` | `{ from, sdp, fromChannelId }` | WebRTC Offer |
| `private-voice-signal-{userId}` | `rtc-answer` | `{ from, sdp }` | WebRTC Answer |
| `private-voice-signal-{userId}` | `rtc-ice-candidate` | `{ from, candidate }` | ICE Candidate |
| `presence-voice-room-{roomId}` | `room-closed` | `{ reason }` | 房间关闭通知 |

### 组件职责说明

#### VoiceRoomPanel（语音房间主面板）
- 以右侧抽屉面板形式呈现
- 三区域布局：顶部标题栏 + 中间内容区（左频道列表 + 右成员列表）+ 底部控制栏
- 支持拖拽改变面板宽度
- 移动端全屏展示

#### VoiceChannelList（频道列表）
- 显示当前房间所有频道
- 每个频道显示名称和在线人数
- 当前所在频道高亮
- 空频道灰色半透明
- 支持频道排序（拖拽）

#### VoiceMemberList（成员列表）
- 显示当前频道内的在线成员
- 每个成员显示头像、名称、角色标签（房主/成员）
- 语音状态实时指示（说话中=绿色波纹、静音=红色标记、闭麦=灰色）
- 支持右键菜单（查看资料、私聊、踢出）

#### VoiceControls（底部控制栏）
- 麦克风开关（toggle 按钮，有状态指示）
- 扬声器静音开关
- 音量调节滑块
- 离开房间按钮（二次确认）
- 设置按钮（音频设备选择、降噪开关）

#### VoiceRoomFloating（全局浮动窗）
- 当用户关闭面板但未离开房间时，显示浮动小窗
- 显示房间名称 + 当前频道 + 通话时长
- 支持拖拽移动位置
- 点击展开回到完整面板
- 始终悬浮在其他 UI 之上（z-index 最高）

## 核心交互流程

### 流程 A：下单后自动进入语音房间

```
用户下单 → 陪玩接单 → 支付完成 → 订单状态变为 in_progress
    ↓
订单详情页出现 [🎤 进入语音房间] 按钮
    ↓
点击 → POST /api/voice-rooms (type="order", orderId=xxx)
    ↓
后端逻辑：
  - 如果该订单已有活跃房间 → 直接返回房间信息
  - 否则创建新房间，默认包含：
    - "大厅"频道
    - 按订单游戏名创建的专属频道
    - "闲聊"频道
  - 将下单用户和陪玩自动加入房间
  - 通知双方加入
    ↓
前端展开 VoiceRoomPanel，用户进入"大厅"频道
    ↓
Pusher 广播 member-added 事件 → 所有在线成员 UI 更新
    ↓
自动建立 WebRTC P2P 连接 → 开始实时语音通话
```

### 流程 B：从俱乐部进入语音房间

```
俱乐部详情页 → 成员操作区 [🎤 进入语音大厅] 按钮
    ↓
GET /api/voice-rooms?clubId=xxx&status=active
    ↓
如果已有活跃房间：
  → POST /api/voice-rooms/{id}/join（加入已有房间）
如果无活跃房间：
  → POST /api/voice-rooms (type="club", clubId=xxx)
  → 自动创建持久化房间，默认频道：
    - "大厅"
    - "陪玩推荐"（特色频道，可浏览在线陪玩）
    - 按俱乐部关联游戏创建对应频道
    - "闲聊"
    ↓
进入房间后：
  "陪玩推荐"频道中可看到俱乐部内的在线陪玩列表
  点击陪玩 → 弹出资料卡 → 可发起私聊或直接下单
  其他频道为自由交流区
```

### 多频道切换流程

```
用户在频道 A → 点击频道列表中的频道 B
    ↓
POST /api/voice-channels/{channelBId}/join
    ↓
前端逻辑：
  1. 关闭与频道 A 中所有 peer 的 WebRTC 连接
  2. 发送 channel-switch 事件（Pusher）
  3. 获取频道 B 的在线成员列表
  4. 与频道 B 中每个成员建立新的 WebRTC 连接
  5. 更新本地状态（currentChannelId）
    ↓
其他成员 UI 更新：
  - 频道 A 在线人数 -1
  - 频道 B 在线人数 +1
  - 成员列表中移除/新增该用户
```

### WebRTC 连接生命周期

```
加入频道
  ↓
获取频道在线成员列表
  ↓
对每个成员：
  1. 创建 RTCPeerConnection
  2. 获取本地音频流 (getUserMedia)
  3. 添加 track 到 peer connection
  4. 创建 Offer → 通过 Pusher 发送
  5. 等待 Answer → 设置 remote description
  6. ICE 协商 → 建立 P2P 连接
  ↓
通话中：
  - AudioContext 实时音量检测 → 更新说话状态
  - 响应 mute/deafen 操作
  - 响应新成员加入（建立新连接）
  - 响应成员离开（关闭连接）
  ↓
离开频道：
  1. 关闭所有 RTCPeerConnection
  2. 停止本地音频流
  3. 发送离开事件（Pusher）
```

## 关键 UI/UX 原则

基于 Web Design Guidelines 对齐：

### 1. 状态可见性
- 每个成员的语音状态实时可见：
  - 说话中 = 绿色光环 + 声波动画
  - 麦克风静音 = 红色静音图标
  - 扬声器静音（闭麦） = 灰色耳机图标
  - 离线 = 灰色头像
- 频道在线人数实时更新（Pusher 推送）

### 2. 渐进式呈现
- 先展示频道列表，点击频道后才展示成员列表
- 控制栏默认收起音量滑块，点击"音量"图标才展开
- 避免初次进入时信息过载

### 3. 操作可达性
- 底部控制栏（麦克风/静音/离开）固定在视口底部
- 移动端拇指可达区域放置关键操作
- 所有点击目标最小 44x44px（符合 WCAG 触摸目标标准）

### 4. 防误触设计
- 离开房间按钮需要二次确认弹窗
- 踢出成员需要确认
- 关闭房间（房主操作）需要输入确认

### 5. 即时反馈
- 说话状态使用 AudioContext API 实时音量检测（≤50ms 延迟）
- 按钮 toggle 状态即时切换，带微动画过渡
- 连接中/连接失败状态明确指示

### 6. 无干扰通知
- 语音通话期间收到新消息不弹窗打断，改用轻量顶部 toast（3s 自动消失）
- 新成员加入房间不播放提示音，仅 UI 更新（避免频繁打断）
- 被邀请加入房间时播放一次提示音 + 弹窗

### 7. 性能优化
- 进入频道时才建立 WebRTC 连接（懒连接）
- 切换频道时立即断开旧频道所有 peer 连接
- 音频流使用 Opus 编码（低带宽消耗）
- 静音时暂停音频轨道发送（节省带宽）
- 组件使用 React.memo 减少不必要渲染

### 8. 无障碍
- 所有按钮有明确的 aria-label
- 语音状态指示器有色盲友好方案（图标 + 颜色双通道）

## 补充说明

### 房间生命周期

```
创建 → 活跃 → 空闲（所有人离开） → 自动关闭 / 手动关闭
                                    ↓
                            order 类型：下单后 X 小时自动过期
                            club 类型：持久存在，管理员手动关闭
```

### 安全考虑

- Pusher presence channel 需要服务端认证，确保只有平台用户能加入
- 信令 API 需验证用户身份 + 房间成员身份
- 房间操作（创建/关闭/踢人）需验证权限（房主/管理员）
- 敏感操作（踢人/关闭房间）记录操作日志

### 兼容性

- WebRTC 在主流浏览器均支持：Chrome 60+, Firefox 55+, Safari 11+, Edge 79+
- 移动端 Safari（iOS 11+）、Chrome（Android）支持良好
- getUserMedia 需要 HTTPS（本地开发 localhost 豁免）
- iOS Safari 需要在用户手势下获取音频权限（点击按钮触发）
